## 1. 项目概述

本工具使用 Go 语言编写，用于将游戏策划编辑的 Excel 数值配置表（`.xlsx`）批量导出为标准 JSON 文件，供客户端或服务端程序读取使用。

核心目标：

- **策划友好**：Excel 表格直观编辑，支持中文字段描述、类型声明、备注列
- **结构丰富**：支持基本类型、集合、Map、嵌套 Struct 等多种字段类型
- **多表合并**：同一 Class 可分散在多个 Sheet 中定义，工具自动合并
- **数据校验**：导出时自动检查 ID 唯一性、类型合法性，报错定位到行列
- **工程化**：命令行驱动，支持批量处理，易于集成到构建流程

## 2. 设计决策汇总

| 决策项 | 选择 | 说明 |
|--------|------|------|
| 输入格式 | Excel (.xlsx) | 使用 excelize 库解析 |
| 表头结构 | 第1行中文含义 / 第2行类型 / 第3行 FieldName | 第4行起为数据行 |
| ClassName 定义 | Sheet 的 A1 单元格 | Sheet 名仅作备注，不参与逻辑 |
| 多 Sheet 合并 | A1 ClassName 相同则合并 | 合并时按 Class 配置的主键规则校验唯一性 |
| 嵌套结构 | 同 Sheet 内特殊列名标记 | 如 `rewards[0].id`、`rewards[0].count` |
| JSON 顶层结构 | 数组 `[]`（Class）/ 对象 `{}`（GlobalConfig） | Class 模式统一输出平铺数组；GlobalConfig 输出 key-value 对象 |
| 忽略列标记 | FieldName 以 `#` 开头 | 该列整体不导出到 JSON |
| 数据校验 | 按 Class 配置的主键规则 + 类型合法性检查 | 校验失败时报错并终止导出 |
| 表结构模式声明 | 每个 Excel 文件内的 `__ClassConfig` Sheet | 支持单主键、联合主键、无主键三种模式 |
| 未声明的 Class | 默认单主键模式 | pk 字段名默认为 `id` |
| 公共配置标记 | A1 以 `!` 开头，如 `!GlobalConfig` | 走独立解析流程，不进入 Class 的 schema 解析 |
| 公共配置类型声明 | 行级（C 列），支持自动推断 | C 列为空时按值特征自动推断；复杂类型需手动填写 |

## 3. Excel 表格规范

### 3.1 Sheet 结构

每个 Sheet 对应一个 Class 的数据分片，**A1 单元格写明 ClassName**。同一 ClassName 的多个 Sheet 会在导出时自动合并为一个 JSON 文件。

| 单元格/行 | 内容 | 示例 |
|-----------|------|------|
| A1 | ClassName（唯一标识，决定输出文件名） | `ItemConfig` |
| 第 1 行（B1起） | 字段中文含义（人类可读注释） | 道具ID、道具名称、出售价格 |
| 第 2 行（B2起） | 字段类型 | `int`、`string`、`[]int`、`map<int,int>` |
| 第 3 行（B3起） | FieldName（英文字段名） | `id`、`name`、`price` |
| 第 4 行起 | 数据行 | 1001、铁剑、100 |

> A1 以 `!` 开头的 Sheet 为公共配置表，走独立解析流程，详见第 5 节。

### 3.2 字段类型系统

第 2 行（类型行）支持以下类型声明：

| 类型写法 | Go 解析结果 | JSON 示例 | 说明 |
|----------|------------|-----------|------|
| `int` | `int64` | `1001` | 整数 |
| `float` | `float64` | `3.14` | 浮点数 |
| `string` | `string` | `"铁剑"` | 字符串 |
| `bool` | `bool` | `true` / `false` | 布尔值 |
| `[]int` | `[]int64` | `[1,2,3]` | 整数数组，单元格用逗号分隔 |
| `[]float` | `[]float64` | `[1.5,2.0]` | 浮点数组，逗号分隔 |
| `[]string` | `[]string` | `["a","b"]` | 字符串数组，逗号分隔 |
| `map<int,int>` | `map[int64]int64` | `{"1":100}` | 键值对，单元格格式 `k:v,k:v` |
| `map<int,string>` | `map[int64]string` | `{"1":"a"}` | 键值对 |
| `map<string,int>` | `map[string]int64` | `{"k":1}` | 键值对 |
| `map<string,string>` | `map[string]string` | `{"k":"v"}` | 键值对 |
| `struct` | object | `{...}` | 嵌套对象，配合列名标记使用 |
| `[]struct` | array of object | `[{...}]` | 对象数组，配合列名标记使用 |
| `map<int,struct>` | map of object | `{"1":{...}}` | 对象 Map，配合列名标记使用 |
| `ignore` | — | — | 忽略该列，与 `#` 前缀等效 |

### 3.3 嵌套结构列名规范

当字段涉及 struct / []struct / map<int,struct> 时，通过列名的特殊语法描述子字段路径：

| FieldName 写法 | 对应结构 | 说明 |
|----------------|----------|------|
| `rewards[0].id` | `rewards[0].id` | `rewards` 是 `[]struct`，第0个元素的 `id` 字段 |
| `rewards[0].count` | `rewards[0].count` | 同上，`count` 字段 |
| `rewards[1].id` | `rewards[1].id` | 第1个元素（固定长度数组展开方式） |
| `attrs.atk` | `attrs.atk` | `attrs` 是 `struct`，`atk` 子字段 |
| `bonus{1}.value` | `bonus{1}.value` | `bonus` 是 `map<int,struct>`，key=1 的 `value` |

解析器将同一前缀（如 `rewards`）的所有列自动组合为嵌套对象或数组。

### 3.4 忽略列规范

以下两种方式均可标记一列为忽略列，导出时该列完全跳过：

- FieldName（第3行）以 `#` 开头，例如：`#备注`、`#策划用`、`#TODO`
- 类型行（第2行）填写 `ignore`

两种方式效果相同，推荐使用 `#` 前缀，视觉上更直观。

### 3.5 表格示例

一个完整的 `ItemConfig` Sheet 示例：

```
A1: ItemConfig

     | B        | C        | D      | E               | F                | G      |
行1  | 道具ID   | 道具名称  | 出售价格 | 奖励[0]物品ID   | 奖励[0]数量      | #策划备注 |
行2  | int      | string   | int    | int             | int              | ignore |
行3  | id       | name     | price  | rewards[0].id   | rewards[0].count | #remark|
行4  | 1001     | 铁剑     | 100    | 2001            | 1                | 测试用  |
行5  | 1002     | 精钢剑   | 500    | 2002            | 2                | 待确认  |
```

对应输出 JSON（`ItemConfig.json`）：

```json
[
  {
    "id": 1001,
    "name": "铁剑",
    "price": 100,
    "rewards": [{ "id": 2001, "count": 1 }]
  },
  {
    "id": 1002,
    "name": "精钢剑",
    "price": 500,
    "rewards": [{ "id": 2002, "count": 2 }]
  }
]
```

## 4. Class 配置表 `__ClassConfig`

### 4.1 作用

`__ClassConfig` 是每个 Excel 文件的元信息表，用于声明本文件内各 Class 的**表结构模式**（主键类型、主键字段、排序字段等）。工具在处理业务 Sheet 之前，会优先读取同文件内的 `__ClassConfig`。

没有 `__ClassConfig` Sheet 的 Excel 文件完全合法，其中所有 Class 均使用默认规则（单主键，pk 字段名为 `id`）。

### 4.2 Sheet 结构

`__ClassConfig` 本身也遵循标准表头规范（第1行中文含义、第2行类型、第3行 FieldName），A1 固定写 `__ClassConfig`。

```
A1: __ClassConfig

     | B          | C                    | D                      | E           | F              | G
行1  | Class名称  | 主键类型              | 主键字段                | 排序字段     | SheetName注入字段 | 注入字段类型
行2  | string     | string               | string                  | string      | string         | string
行3  | className  | pkType               | pkFields                | sortFields  | sheetNameAs    | sheetNameType
行4  | ItemConfig | single               | id                      |             |                |
行5  | EquipSlot  | composite            | equipTypeId,slotId      |             |                |
行6  | RandomReward| none                |                         | groupId     | groupId        | int
```

### 4.3 字段说明

| FieldName | 类型 | 必填 | 说明 |
|-----------|------|------|------|
| `className` | string | 是 | 必须与对应 Sheet 的 A1 完全一致 |
| `pkType` | string | 是 | 取值见下表 |
| `pkFields` | string | 条件必填 | `single`/`composite` 时必填，逗号分隔多个字段名 |
| `sortFields` | string | 否 | `none` 模式下指定排序列，逗号分隔；不填则保持 Excel 行顺序 |
| `sheetNameAs` | string | 否 | 将 SheetName 的值注入为指定字段名，如 `groupId` |
| `sheetNameType` | string | 条件必填 | `sheetNameAs` 填写时必填，指定注入字段的类型，取值与字段类型系统一致（如 `int`、`string`） |

**pkType 取值：**

| 值 | 含义 | 唯一性校验 | 排序规则 |
|----|------|------------|----------|
| `single` | 单主键 | pkFields 指定的单列值全局唯一 | 按主键列升序 |
| `composite` | 联合主键 | pkFields 指定的多列组合值全局唯一 | 保持 Excel 行顺序 |
| `none` | 无主键 | 不做唯一性校验 | 按 sortFields 升序；不填则保持 Excel 行顺序 |

**sheetNameAs 注入规则：**

- 注入字段不需要在业务 Sheet 的表头里定义，由工具在 builder 阶段自动追加到每行数据末尾
- 注入字段按 `sheetNameType` 做类型转换，转换失败则报错（如 SheetName 为 `"abc"`，但类型声明为 `int`）
- 注入字段可以作为 `pkFields` 或 `sortFields` 使用
- 注入字段在 JSON 中的位置固定在所有普通字段之后

### 4.4 三种模式完整示例

**模式一：单主键（ItemConfig）**

```
A1: ItemConfig
     | B      | C      | D
行1  | 道具ID  | 道具名称 | 出售价格
行2  | int    | string  | int
行3  | id     | name    | price
行4  | 1001   | 铁剑    | 100
行5  | 1002   | 精钢剑  | 500
```

输出 `ItemConfig.json`（按 id 升序）：
```json
[
  { "id": 1001, "name": "铁剑",  "price": 100 },
  { "id": 1002, "name": "精钢剑", "price": 500 }
]
```

**模式二：联合主键（EquipSlotConfig）**

```
A1: EquipSlotConfig
     | B           | C       | D         | E
行1  | 装备分类ID   | 装备部位ID | 攻击力加成 | 血量加成
行2  | int         | int     | float     | float
行3  | equipTypeId | slotId  | atkBonus  | hpBonus
行4  | 1           | 1       | 0.05      | 0.0
行5  | 1           | 2       | 0.0       | 0.10
行6  | 2           | 1       | 0.08      | 0.0
```

输出 `EquipSlotConfig.json`（保持 Excel 行顺序）：
```json
[
  { "equipTypeId": 1, "slotId": 1, "atkBonus": 0.05, "hpBonus": 0.0  },
  { "equipTypeId": 1, "slotId": 2, "atkBonus": 0.0,  "hpBonus": 0.10 },
  { "equipTypeId": 2, "slotId": 1, "atkBonus": 0.08, "hpBonus": 0.0  }

```

**模式三：无主键 + SheetName 注入（RandomRewardPool）**

`__ClassConfig` 配置：`sheetNameAs=groupId`，`sheetNameType=int`，`sortFields=groupId`

Excel 文件包含两个业务 Sheet，**Sheet 名就是 groupId 的值**，表头里不需要定义 groupId 列：

```
Sheet名: 1001

     | B       | C      | D       | E      | F
行1  | 奖励类型 | 奖励Code | 奖励数量 | 权重   | 最低等级条件
行2  | int     | string  | int     | int    | int
行3  | type    | code    | amount  | weight | minLevel
行4  | 1       | item_001 | 1      | 100    | 1
行5  | 1       | item_002 | 2      | 50     | 10
```

```
Sheet名: 1002
A1: RandomRewardPool

     | B       | C      | D       | E      | F
行1  | 奖励类型 | 奖励Code | 奖励数量 | 权重   | 最低等级条件
行2  | int     | string  | int     | int    | int
行3  | type    | code    | amount  | weight | minLevel
行4  | 1       | item_003 | 1      | 200    | 1
行5  | 1       | item_004 | 1      | 200    | 1
```

输出 `RandomRewardPool.json`（builder 自动将 SheetName 转为 int 注入为 groupId，按 groupId 升序）：
```json
[
  { "type": 1, "code": "item_001", "amount": 1, "weight": 100, "minLevel": 1,  "groupId": 1001 },
  { "type": 1, "code": "item_002", "amount": 2, "weight": 50,  "minLevel": 10, "groupId": 1001 },
  { "type": 1, "code": "item_003", "amount": 1, "weight": 200, "minLevel": 1,  "groupId": 1002 },
  { "type": 1, "code": "item_004", "amount": 1, "weight": 200, "minLevel": 1,  "groupId": 1002 }

```

### 4.5 多 Sheet 合并与 __ClassConfig 的关系

- 同一 xlsx 文件内，或跨不同 xlsx 文件，只要 A1 ClassName 相同，均视为同一 Class 的分片
- **字段定义可以跟第一个被解析到的 Sheet 不同**，不需要后续 Sheet 保持跟第一个 Class Sheet 一致
- 合并后按 ClassMeta 的 pkType 执行唯一性校验
- ClassMeta 以第一个 Sheet 所在文件的 `__ClassConfig` 为准；若后续文件对同一 ClassName 有不同配置，报错

## 5. 公共配置表 `!GlobalConfig`

### 5.1 作用与适用场景

公共配置（GlobalConfig）用于存放**全局单值型配置项**，每一行是一条独立的配置，不同行的值类型可以不同。典型场景包括：

| 场景 | 示例 |
|------|------|
| 纯数值 | 装备最大升级上限：`5` |
| 纯字符串（含公式） | 将领升级消耗公式：`Math.ceil(200*(1+(level-1)*0.005))` |
| 整数列表 | 所有怪兽默认携带的被动技能 ID：`100001,100002` |
| 字符串到字符串的 Map | 首次出手时间规则：`general:INSTANT_FIRST,role:INSTANT_FIRST` |
| 字符串到数值的 Map | 基础属性加成：`att:10,hpMax:200` |

公共配置与 Class 配置的核心区别在于：**类型声明是行级的，而非列级的**。因此工具对 `!GlobalConfig` Sheet 走完全独立的解析分支。

### 5.2 Sheet 结构

A1 单元格以 `!` 开头（如 `!GlobalConfig`），工具据此识别为公共配置表，不进入 Class 的 schema 解析流程。

```
A1: !GlobalConfig

     | B                                          | C                  | D
行1  | 唯一标识                                    | 值类型              | 内容
行2  | string                                     | string             | string
行3  | id                                         | type               | value
行4  | battle:maxEquipLevel                       |                    | 5
行5  | battle:generalLevelCostFormula             |                    | Math.ceil(200*(1+(level-1)*0.005))
行6  | battle:monsterPassiveSkillIds              | []int              | 100001,100002
行7  | battle:firstAttackIntervalStrategyMap      | map<string,string> | general:INSTANT_FIRST,role:INSTANT_FIRST,monster:INSTANT_FIRST
行8  | battle:baseAttrBonus                       | map<string,int>    | att:10,hpMax:200
行9  | battle:enableNewBattleSystem               |                    | true
```

**列说明：**

| 列 | FieldName | 类型 | 必填 | 说明 |
|----|-----------|------|------|------|
| B | `id` | string | 是 | 配置项唯一标识，全局不可重复 |
| C | `type` | string | 否 | 值类型声明；为空时触发自动推断（见 5.3 节） |
| D | `value` | — | 是 | 配置项的值，按 C 列声明（或自动推断）的类型解析 |

> 行2 的类型行对 D 列固定填 `string` 作占位，工具在处理 `!GlobalConfig` 时忽略对 D 列的静态类型校验，改为按每行 C 列做动态解析。

### 5.3 C 列类型自动推断规则

当 C 列（type）为空时，工具对 D 列原始字符串按以下三层规则依次判断：

**第一层：数值与布尔推断（静默，无 WARN）**

| 推断条件 | 推断结果 | 示例 |
|----------|----------|------|
| 能被 `strconv.ParseInt` 解析 | `int` | `5`、`-10`、`0` |
| 能被 `strconv.ParseFloat` 解析，**且值含小数点** | `float` | `3.14`、`0.5` |
| 值为 `true` / `false`（大小写不敏感） | `bool` | `true`、`FALSE` |

> **float 推断说明**：必须含小数点才推断为 `float`，否则一律推断为 `int`。例如 `5` → `int`，`5.0` → `float`。如需明确输出 float，请在 C 列显式填写 `float`。

**第二层：复杂类型特征检测（降级为 string，打印 WARN）**

若 D 列值含有 `,` 或 `:` 等复杂结构特征，但 C 列未显式声明类型，则：
- 工具将该值降级输出为 `string`
- 同时打印 WARN，提示策划可能需要手动补填 C 列类型

```
[WARN] GlobalConfig.xlsx / Sheet1 / 行6 / 列D (value): 值 "100001,100002" 含逗号分隔符，
       已自动推断为 string，若意图为 []int 请在 C 列显式填写类型
[WARN] GlobalConfig.xlsx / Sheet1 / 行7 / 列D (value): 值 "general:INSTANT_FIRST,..." 含冒号与逗号，
       已自动推断为 string，若意图为 map<string,string> 请在 C 列显式填写类型
```

**第三层：其余所有情况（推断为 string，静默）**

不含上述特征的值，静默推断为 `string`，无需任何提示。

**推断优先级总结：**

```
C 列已填写 → 直接使用
C 列为空   → ParseInt 成功       → int
           → ParseFloat 且含'.' → float
           → 值为 true/false    → bool
           → 含 ',' 或 ':'      → string + WARN
           → 其余              → string（静默）
```

### 5.4 支持的 C 列类型声明

C 列可以填写的类型与 Class 字段类型系统完全一致（见 3.2 节），工具复用同一套类型转换逻辑。常用类型参考：

| C 列填写 | D 列示例值 | JSON 输出 |
|----------|-----------|-----------|
| `int` | `5` | `5` |
| `float` | `3.14` | `3.14` |
| `bool` | `true` | `true` |
| `string` | `Math.ceil(...)` | `"Math.ceil(...)"` |
| `[]int` | `100001,100002` | `[100001, 100002]` |
| `[]string` | `a,b,c` | `["a","b","c"]` |
| `map<string,string>` | `general:INSTANT_FIRST,role:INSTANT_FIRST` | `{"general":"INSTANT_FIRST","role":"INSTANT_FIRST"}` |
| `map<string,int>` | `att:10,hpMax:200` | `{"att":10,"hpMax":200}` |

### 5.5 输出格式

公共配置输出为 `GlobalConfig.json`，顶层结构为 **JSON 对象 `{}`**（以 id 为 key，便于消费侧按 key 直接查值）。

以 5.2 节示例为例，输出结果为：

```json
{
  "battle:maxEquipLevel": 5,
  "battle:generalLevelCostFormula": "Math.ceil(200*(1+(level-1)*0.005))",
  "battle:monsterPassiveSkillIds": [100001, 100002],
  "battle:firstAttackIntervalStrategyMap": {
    "general": "INSTANT_FIRST",
    "role": "INSTANT_FIRST",
    "monster": "INSTANT_FIRST"
  },
  "battle:baseAttrBonus": {
    "att": 10,
    "hpMax": 200
  },
  "battle:enableNewBattleSystem": true
}
```

> 输出键的顺序与 Excel 行顺序一致。

### 5.6 多 Sheet / 多文件合并规则

- 不同 xlsx 文件或同一 xlsx 文件内，A1 为 `!GlobalConfig` 的所有 Sheet 均合并到同一份 `GlobalConfig.json`
- 合并时对 B 列（id）做全局唯一性校验，重复则报错并终止导出
- 不同 Sheet 可以使用不同的 id 前缀（如 `battle:`、`lobby:`）作为命名空间隔离，工具不做强制约束

### 5.7 校验规则

| 校验项 | 触发条件 | 处理方式 |
|--------|----------|----------|
| id 唯一性 | B 列出现重复 id | 报错并终止 |
| C 列类型非法 | C 列填写了不支持的类型字符串 | 报错并终止 |
| D 列值类型转换失败 | D 列值无法按 C 列声明的类型解析 | 报错并终止 |
| C 列为空且 D 列含复杂特征 | 含 `,` 或 `:` 但未声明类型 | 降级为 string 并打印 WARN |
| B 列（id）为空 | 某行 B 列无值 | 报错并终止 |

错误信息格式示例：

```
[ERROR] GlobalConfig.xlsx / Sheet1 / 行6 / 列D (value): 期望 []int 类型，实际值 "100001,abc" 解析失败
[ERROR] GlobalConfig.xlsx / Sheet1 / 行9 / 列B (id): id 重复，值 "battle:maxEquipLevel" 已在行4出现
[WARN]  GlobalConfig.xlsx / Sheet1 / 行7 / 列D (value): 值含冒号与逗号，已自动推断为 string，若意图为 map 类型请在 C 列显式填写
```

### 5.8 与 Class 模式的对比

| 维度 | Class 模式 | GlobalConfig 模式 |
|------|-----------|------------------|
| A1 标记 | 普通 ClassName，如 `ItemConfig` | `!` 前缀，如 `!GlobalConfig` |
| 类型声明位置 | 第2行（列级，整列统一类型） | 每行 C 列（行级，每行独立类型） |
| 输出顶层结构 | 数组 `[]` | 对象 `{}` |
| 主键 / 唯一键 | 由 `__ClassConfig` 的 pkFields 配置 | B 列（id）固定为唯一键 |
| 类型推断 | 无（必须显式声明） | 支持（int / float / bool / string 自动推断） |
| 新增模块 | — | `globalconfig/parser.go` |

### 6.1 整体流程

                                        ▼

### 6.3 核心数据结构

```go
// classconfig.PkType — 主键类型枚举
type PkType string
const (
    PkTypeSingle    PkType = "single"
    PkTypeComposite PkType = "composite"
    PkTypeNone      PkType = "none"
)

// classconfig.ClassMeta — 一个 Class 的元配置（来自 __ClassConfig）
type ClassMeta struct {
    ClassName     string   // Class 名称
    PkType        PkType   // 主键类型
    PkFields      []string // 主键字段名列表（single 时长度为1，composite 时长度>=2）
    SortFields    []string // 排序字段（none 模式下使用）
    SheetNameAs   string   // 将 SheetName 注入为该字段名（空字符串表示不注入）
    SheetNameType string   // SheetNameAs 对应的类型声明（如 "int"、"string"）
}

// schema.FieldDef — 一个字段的元信息
type FieldDef struct {
    Label     string    // 中文含义（第1行）
    TypeStr   string    // 原始类型字符串（第2行）
    FieldName string    // 英文字段名（第3行）
    FieldType FieldType // 解析后的枚举类型
    ColIndex  int       // 列索引，用于错误定位
    Ignored   bool      // 是否忽略
}

// schema.SheetSchema — 一个 Sheet 解析出的元信息
type SheetSchema struct {
    ClassName string
    Fields    []FieldDef
    DataRows  [][]string // 原始字符串数据（第4行起）
    FileName  string     // 来源文件名，用于错误定位
    SheetName string     // 来源 Sheet 名，用于错误定位
}

// merger.ClassData — 合并后的一个 Class 完整数据
type ClassData struct {
    ClassName string
    Meta      classconfig.ClassMeta  // 来自 __ClassConfig，未声明时使用默认值
    Schema    schema.SheetSchema     // 字段定义以第一个 Sheet 为准
    Rows      [][]string             // 合并后的所有数据行
}

// globalconfig.GlobalEntry — 公共配置中的一条配置项
type GlobalEntry struct {
    ID       string      // B 列：唯一标识
    TypeStr  string      // C 列：类型声明（空字符串表示需自动推断）
    RawValue string      // D 列：原始字符串值
    Value    interface{} // 按 TypeStr 解析后的值
    RowIndex int         // 行索引，用于错误定位
}
```

## 7. 关键逻辑说明

### 7.1 嵌套列名解析算法

`builder` 模块在处理 `rewards[0].id` 这类列名时，采用**路径写入策略**：

1. 将列名按 `.` 分割为路径段，如 `["rewards[0]", "id"]`
2. 识别路径段中的 `[N]` 后缀 → 数组索引；`{K}` 后缀 → map key
3. 按路径段逐层在目标 map 中创建或访问节点
4. 将该列的值写入叶子节点

示例：`rewards[0].id = 2001` 的写入等价于 `row["rewards"][0]["id"] = 2001`。

路径段语法规则：

| 写法 | 含义 |
|------|------|
| `fieldName` | 普通对象字段 |
| `fieldName[N]` | 数组第 N 个元素（0-based） |
| `fieldName{K}` | map 中 key 为 K 的元素 |

### 7.2 多 Sheet 合并规则

- 同一 xlsx 文件内，或跨不同 xlsx 文件，只要 A1 ClassName 相同，均视为同一 Class 的分片
- **字段定义可以跟第一个被解析到的 Sheet 不同**，不需要后续 Sheet 保持跟第一个 Class Sheet 一致
- 合并后按 ClassMeta 的 pkType 执行唯一性校验（见下）
- ClassMeta 以第一个 Sheet 所在文件的 `__ClassConfig` 为准；若后续文件对同一 ClassName 有不同配置，报错

### 7.3 唯一性校验规则

| pkType | 校验逻辑 |
|--------|----------|
| `single` | pkFields[0] 对应列的值，在合并后的所有行中全局唯一 |
| `composite` | pkFields 各列的值组合成 `"v1|v2|..."` 字符串，全局唯一 |
| `none` | 不做任何唯一性校验 |

### 7.4 排序规则

| pkType | sortFields 是否填写 | 排序行为 |
|--------|---------------------|----------|
| `single` | — | 按主键列值升序（数值类型按数值比较，字符串按字典序） |
| `composite` | — | 保持 Excel 合并后的行顺序，不排序 |
| `none` | 填写了 | 按 sortFields 各列依次升序 |
| `none` | 未填写 | 保持 Excel 合并后的行顺序，不排序 |

### 7.5 SheetName 注入逻辑

当 ClassMeta 配置了 `sheetNameAs` 时，builder 在处理每一行数据时额外执行以下步骤：

1. 取当前行所属 Sheet 的 SheetName（即 Excel 的标签名）
2. 按 `sheetNameType` 做类型转换（规则与普通字段类型转换相同）
3. 将转换后的值以 `sheetNameAs` 为 key 写入该行的输出 map
4. 注入字段追加在所有普通字段之后

注意：`sheetNameAs` 指定的字段名不能与业务 Sheet 表头中已有的 FieldName 重复，否则报错。

### 7.6 GlobalConfig 行级类型推断逻辑

`globalconfig` 模块对每一行的 C 列（type）和 D 列（value）执行以下推断流程：

```
if C列不为空:
    使用 C 列声明的类型，复用 builder 的类型转换逻辑解析 D 列值
else:
    rawVal = D 列原始字符串
    if strconv.ParseInt(rawVal) 成功:
        推断为 int，静默
    else if strconv.ParseFloat(rawVal) 成功 且 rawVal 含 '.':
        推断为 float，静默
    else if strings.ToLower(rawVal) in {"true", "false"}:
        推断为 bool，静默
    else if rawVal 含 ',' 或 ':':
        降级为 string，打印 WARN 提示补填 C 列
    else:
        推断为 string，静默
```

### 7.7 类型转换规则

| 类型 | 单元格原始值 | 转换逻辑 |
|------|------------|----------|
| `int` | `"1001"` | `strconv.ParseInt`，失败则报错 |
| `float` | `"3.14"` | `strconv.ParseFloat` |
| `bool` | `"true"` / `"1"` / `"是"` | 大小写不敏感，`"1"` / `"是"` 均视为 `true` |
| `[]int` | `"1,2,3"` | 按逗号 split 后逐个 ParseInt |
| `[]string` | `"a,b,c"` | 按逗号 split |
| `map<string,string>` | `"k1:v1,k2:v2"` | 按逗号 split，再按冒号 split 为 k/v |
| `map<string,int>` | `"att:10,hpMax:200"` | 按逗号 split，再按冒号 split，值做 ParseInt |
| `map<int,int>` | `"1:100;2:200"` | 按分号 split，再按冒号 split 为 k/v |
| struct 路径列 | 普通标量值 | 由路径写入算法组装到父对象中 |
| 空单元格 | `""` | 数值类型输出 `0`，字符串输出 `""`，数组输出 `[]` |

## 9. 错误处理规范

所有错误信息须定位到具体位置，格式为：

```
[ERROR] ItemConfig.xlsx / Sheet2 / 行6 / 列D (price): 期望 int 类型，实际值 "abc"
[ERROR] EquipSlotConfig.xlsx / Sheet1 / 行8 / (equipTypeId,slotId): 联合主键重复，值 (2,1) 已在行6出现
[ERROR] GlobalConfig.xlsx / Sheet1 / 行9 / 列B (id): id 重复，值 "battle:maxEquipLevel" 已在行4出现
[ERROR] GlobalConfig.xlsx / Sheet1 / 行6 / 列D (value): 期望 []int 类型，实际值 "100001,abc" 解析失败
[WARN]  SkillConfig.xlsx / Sheet3: A1 为空，跳过该 Sheet
[WARN]  GlobalConfig.xlsx / Sheet1 / 行7 / 列D (value): 值 "general:INSTANT_FIRST,..." 含冒号与逗号，已自动推断为 string，若意图为 map 类型请在 C 列显式填写
```

| 错误类型 | 触发条件 | 处理方式 |
|----------|----------|----------|
| A1 为空 | Sheet 的 A1 单元格无 ClassName | 跳过该 Sheet 并打印 WARN |
| 字段定义冲突 | 同 ClassName 的多 Sheet 字段定义不一致 | 报错并终止 |
| ClassMeta 冲突 | 同 ClassName 在多文件 `__ClassConfig` 中配置不一致 | 报错并终止 |
| pkFields 字段不存在 | `__ClassConfig` 中声明的 pkFields 在 Sheet 字段中找不到 | 报错并终止 |
| sheetNameAs 字段名冲突 | `sheetNameAs` 指定的字段名与业务表头已有 FieldName 重复 | 报错并终止 |
| sheetNameAs 类型转换失败 | SheetName 无法转换为 `sheetNameType` 声明的类型 | 报错并终止 |
| 类型转换失败 | 单元格值无法转换为声明的类型 | 报错并终止 |
| 单主键重复 | `single` 模式下主键值重复 | 报错并终止 |
| 联合主键重复 | `composite` 模式下组合键值重复 | 报错并终止 |
| 嵌套路径冲突 | 同一路径被赋值多次且值不同 | 报错并终止 |
| GlobalConfig id 为空 | B 列某行无值 | 报错并终止 |
| GlobalConfig id 重复 | B 列出现重复 id（含跨 Sheet 合并后） | 报错并终止 |
| GlobalConfig 类型非法 | C 列填写了不支持的类型字符串 | 报错并终止 |
| GlobalConfig 值解析失败 | D 列值无法按 C 列声明的类型解析 | 报错并终止 |
| GlobalConfig 复杂值未声明类型 | D 列含 `,` 或 `:` 但 C 列为空 | 降级为 string 并打印 WARN |
| 输出目录不存在 | 输出目录不存在 | 自动创建 |

## 10. 输出文件规范

- 每个 ClassName 输出一个 JSON 文件，文件名为 `<ClassName>.json`
- **Class 模式**：顶层结构为 JSON 数组 `[]`
- **GlobalConfig 模式**：顶层结构为 JSON 对象 `{}`，以 id 为 key
- 排序规则由 ClassMeta 决定（见 7.4 节）；GlobalConfig 保持 Excel 行顺序
- 数值类型（int/float）直接输出为 JSON number，不加引号
- 空单元格：数值类型输出 `0`，字符串输出 `""`，数组输出 `[]`
- 字段顺序与 Excel 列顺序一致
- `__ClassConfig` Sheet 本身不输出为 JSON

输出目录结构示例：

```
output/
  ItemConfig.json
  SkillConfig.json
  QuestConfig.json
  GlobalConfig.json
```
